Why this exists

Apartment societies run on WhatsApp groups and paper registers. A leaking pipe gets reported to whoever's near the office that day, and by the time it's fixed, nobody remembers it was flagged three weeks earlier as overdue.

The actual hard problem isn't the CRUD — it's trust: a resident needs to see their complaint was really acted on, an admin needs every society's data walled off from every other, and a visitor at the gate needs to be checked in without anyone improvising a paper logbook.

This project treats that as an architecture problem, not a UI problem. Every complaint carries a full, immutable status history — timestamp, actor, note — not a single mutable "status" field. Every query is scoped by tenant at the repository layer, not hoped for at the controller. Every role-gated route was checked with a test that actually tries to break it.

The rest of this page is the receipts.

What it does

Ten modules, one platform

Everything a society actually needs day to day — not a feature checklist, a set of workflows real residents and admins use.

Complaint tracking

Category, photo, priority, and a full status-change history — every transition timestamped with who made it and why.

Overdue detection

Computed at query time against a per-society configurable threshold — never stale, never needs a background job to stay correct.

Notice board

Important notices pin to the top and trigger an email to every resident — regular ones just post.

Visitor gate, with QR

A resident approves a visitor, gets a QR pass; the watchman scans it in and out. Admin sees a live gate log.

SOS emergency alert

One tap notifies every admin and watchman instantly, by design — no form to fill in during an emergency.

Online billing

Real Razorpay checkout, with the payment signature verified server-side before anything is marked paid.

Society polls

One vote per resident, enforced at the database level — not a client-side check that a second tap can slip past.

AI-assisted triage

Suggests a category and priority as a resident types — free by default, with a documented upgrade path to a real LLM call.

Admin dashboard

Totals by status and category, overdue count, all computed live — nothing to keep in sync.

True multi-tenancy

Every society gets its own subdomain, branding, and users — isolated by discipline at the repository layer, not by hope.

How it's built

Decisions, not defaults

Four calls that shape everything else, each made for a stated reason.

tenancy

Shared database, shared schema, row-level isolation. Database-per-tenant doesn't scale operationally past a handful of societies; schema-per-tenant is heavy to migrate. One schema, a mandatory tenantId on every query, enforced at the repository layer where it can't be silently skipped.

auth

Short-lived JWT, rotating refresh tokens. The access token can't be revoked early — so it's 15 minutes. The refresh token can, because only its hash is trusted; using one revokes it and issues a new one, making theft-and-reuse detectable.

data

Overdue status is computed, never stored. A stored boolean goes stale the instant the clock passes the threshold, needing a cron job just to stay honest. Computing it at query time against the tenant's live threshold can't drift from the truth.

providers

Storage, email, and AI triage sit behind a Strategy interface. Cloudinary today, S3 later; keyword-based triage today, a real Claude API call later — swapping either is a config change, proven by writing (and keeping) two real implementations of each, not one.

The receipts

Bugs this project's own process actually caught

Every entry below was found by running something — a test, a linter, a real database — not by re-reading code and hoping. That's the difference this page is actually trying to show.

complaints/
repository
FOUND BY AUDIT

"Overdue first" sorting was applied after pagination, so an overdue complaint on page two could never surface to page one — directly contradicting the spec it was built against.

Fix: sort the full result set before slicing into a page, not after.

RBAC /
complaints
FOUND BY AUDIT

Two valid login roles (Watchman, Maintenance Staff) had no route-level gate on the complaints list — the service layer's role check silently treated "not a resident" as "trusted like an admin."

Fix: explicit role gate at the route, confirmed with a regression test that asserts those roles are rejected.

validation
middleware
FOUND BY A FAILING TEST

Schema validation checked coerced types but never wrote them back to the request — ?sortOverdueFirst=false stayed the string "false", which is truthy in JavaScript.

Fix: write the parsed output back onto the request after validation succeeds.

SOS
alerts
FOUND BEFORE SHIPPING

Sorting alerts by status alphabetically put "acknowledged" before "active" — the most urgent, unhandled alerts would have been buried under already-handled ones.

Fix: explicit priority order, not an enum's alphabetical accident.

db init
script
FOUND WHILE RUNNING IT FOR REAL

The script's own error handler swallowed the actual failure message, so a real error would have been completely undebuggable.

Fix: log the real error before exiting — caught only by actually running the script against a live database.

ansible
playbook
FOUND BY STRICT LINT

A deprecated module and two become/changed_when scoping gaps — invisible in a syntax check, flagged at ansible-lint's strictest "production" profile.

Fix: rewritten to the modern pattern — now a clean pass at every profile level.
